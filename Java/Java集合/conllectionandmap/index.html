Java集合容器-快速了解

发表于 2019-04-26 | 更新于: 2019-04-29 | 分类于 Java ， Java集合 | | 热度: ℃

Java 集合容器

Collection 接口：存储对象的集合

Map 接口：存储键值对（俩个对象）的映射表

容器中的设计模式

迭代器模式

  1. Iterable 接口遍历聚合对象
  2. foreach 循环遍历

    适配器模式

    如：java.util.Arrays#asList() 可以把数组类型转换为 List 类型

    public static <T> List<T> asList(T... a)
    asList() 的参数为泛型的变长参数，不能使用基本类型数组作为参数，只能使用相应的包装类型数组。

Collection 接口

List 接口、Set接口、Queue接口

Set

无序性、不可重复、允许null值。

  1. HashSet：基于哈希表实现；
  2. TreeSet：基于红黑树实现，支持有序性操作；
  3. LinkedHashSet：具有 HashSet 的查找效率，且内部使用双向链表维护元素的插入顺序；

Queue

  1. LinkedList：可以用它来实现双向队列。
  2. PriorityQueue：基于堆结构实现，可以用它来实现优先队列

List

有序性、可重复、允许null值

ArrayList

底层是数组，可通过索引访问，因此查询效率高，但添加和删除数据的效率低，会伴随后面数据的移动

底层是动态数组，因此，支持随机访问

扩容

数组默认大小是10，当添加元素时，ensureCapacityInternal() 会保证容量足够；

当不够时，会调用 grow() 进行扩容，新容量是就容量的1.5倍 newCapacity = oldCapacity + oldCapacity >> 1；

当同时添加的元素超过扩容的容量时，会扩容到元素总数的容量，比如同时添加6个元素，Arraylist会扩容到16。
扩容操作时，会将旧数组复制到新数组中，代价很高，因此，最好在创建时就指定好合适的容量大小，减少扩容次数。

删除元素 remove()

删除时，会调用 System.arraycopy() 将 index 后面的元素向前移动，操作的时间复杂度是 O(N)，
因此，ArrayList 删除元素的代价很高。

Fail-Fast

modCount 用来记录 ArrayList 结构发生变化的次数，也就是添加、删除元素或调整内部数组大小的操作次数。

在进行序列化或迭代时，需要比较前后的 modCount 是否改变，如果改变了，会抛出 ConcurrentModificationException。

也就是，在进行序列化或迭代时，不可以进行添加或删除的操作(阿里手册上有相关的描述)。

序列化

由于动态扩容的原因，ArrayList的某些元素并没有被使用，因此无需将每个元素都序列化。

transcation 关键字修饰数组，默认不会被序列化，ArrayList 通过 wirteObject() & readObject() 控制只序列化有数据的元素。

Vector

#####同步
Vector 与 ArrayList 的实现类似，区别是使用了 synchronized 进行了同步。

与 ArrayList 相比

Vector 是线程同步的，开销较大，访问较慢，推荐使用 ArrayList ，同步可以有程序员进行控制。

Vector 每次扩容请求其 2 倍的容量大小，ArrayList 为 1.5 倍。

替代方案
  1. JUC 包下 CopyOnWriteArrayList 类。

    List<String> list = new CopyOnWriteArrayList<>();
  2. 通过 Collections.synchronizedList() 方法将 ArrayList 变为线程安全的。

    List<String> list = new ArrayList<>();

    List<String> synList = Collections.synchronizedList(list);
CopyOnWriteArrayList
读写分离

读操作在原始数据上进行，写操作在一个复制的新数组上进行。

写操作加锁，防止并发写入的数据丢失；写操作结束，将原数组指向新数组。

使用场景

优点：

CopyOnWriteArrayList 在写操作时，可以进行读操作，提高了读操作的性能，适用于读多写少的应用场景。

缺点：

  1. 写操作需要复制新数组，所以内存占用是原来的两倍左右。
  2. 当部分写操作没有同步到数组时，读操作不具有实时性。

LinkedList

基于双向链表实现,Node 存储链表的节点信息，每个链表节点都维护了 first 指针和 last 指针，因此只能顺序访问。

使用场景

可用作栈、队列和双向队列。

与 ArrayList 相比
  1. 底层实现方式不同，A 基于动态数组，L 基于双向链表；
  2. A 支持随机访问，L 不支持；
  3. A 查询效率更高，L 在任意位置添加和删除效率更高；

Map 接口

键唯一、值不唯一

  1. TreeMap：基于红黑树实现。
  2. HashTable：和 HashMap 类似，但它是线程安全的。它是遗留类，不应该去使用它。现在可以使用 ConcurrentHashMap 来支持线程安全，并且 ConcurrentHashMap 的效率会更高，因为 ConcurrentHashMap 引入了分段锁。

HashMap

基于哈希表，默认初始大小为 16；

HashMap 构造函数允许用户传入的容量不是 2^n，因为它可以自动地将传入的容量转换为 2^n。

存储结构

内部包含一个 Entry 数组 transient Entry[] table;
Entry 是一个链表，存储着键值对。包含四个属性 key、value、next、hash；

即数组中的每个位置为一个桶，桶里包含一个链表。

同一个链表存放 hash 值相同的 Entry，HashMap 使用拉链法实现后插入元素在链表头部。

查询步骤

  1. 计算键值的 hashCode ，与桶的个数取余，得到桶下标。如：115 % 16 = 3；
  2. 在链表上顺序查找，时间复杂度显然和链表的长度成正比

    插入 put

    HashMap 可以插入键为 null的键值对，因为 null 无法调用 hashCode() 计算桶下标，因此 HashMap 强制使用第0个桶存放键为 null的键值对。
  3. 计算键值，得到桶下标
  4. 使用链表的头插法，将新元素插入链表头部

    动态扩容

    设 HashMap 的数组容量为 M，需要存储的键值对数量为 N，为提高查找的效率，每个链表的的长度大约为(N/M)，因此查询的复杂度为O(N/M)。

    HashMap 会使用动态扩容，根据当前键值对数量，动态调整 M 值，使得 (N/M) 因可能小，从而降低查询成本。

    扩容相关的参数主要有：capacity、size、threshold 和 load_factor
  • capacity：数组容量，默认16
  • size：键值对数量
  • threshold：size 的临界值，当 size >= threshold 时，必须进行扩容操作
  • loadFactor：装载因子，默认0.75，threshold = capacity * loadFactor

    使用 resize() 进行扩容，没扩容后数组容量为原来的 2 倍；

    扩容操作同样需要把 oldTable 的所有键值对重新插入 newTable 中。

    链表转红黑树

    JDK1.8后，当链表长度 >8 时，链表将转换为红黑树。

    与 HashTable 的比较

  1. HashTable 使用 synchronized 进行线程同步
  2. HashMap 可以存放建为空的键值对

ConcurrentHashMap

ConcurrentHashMap 在 HashMap 的基础上添加了分段锁(Segment)，每个分段锁维护几个桶(HashEntry)，多个线程可以同时访问不同分段锁上的桶，从而使其并发度更高（并发度就是 Segment 的个数）。

Segment 继承自 ReentrantLock。默认的并发级别为 16，也就是说默认创建 16 个 Segment。

size 操作

每个 Segment 维护了一个 count 变量来统计该 Segment 中的键值对个数。执行 size 操作时，需要遍历所有 Segment 然后把 count 累计起来。

在执行 size 操作时先尝试不加锁，如果连续两次不加锁操作得到的结果一致，那么可以认为这个结果是正确的。

如果尝试的次数超过 2 次，就需要对每个 Segment 加锁。

JDK 1.8 的改动

JDK1.7采用分段锁机制来实现并发更新操作，核心类为 Segment，继承自重入锁 ReentrantLock；

JDK 1.8 使用了 CAS 操作来支持更高的并发度，在 CAS 操作失败时使用内置锁 synchronized；且链表过长时会转换为红黑树。

LinkedHashMap

存储结构

基于哈希表，内部使用双向链表维护插入顺序或者 LRU 顺序

LRU 顺序：每次访问一个节点时，将该节点移到链表尾部；从而保证链表尾部是最近访问的节点，链表首部是最近最久未使用的节点。
在 put 等操作之后执行时，会移除链表首部的节点。

使用场景

使用 LinkedHashMap 实现的一个 LRU 缓存。

  1. 设定最大缓存空间，开启 LRU 顺序；
  2. 开启 LRU 顺序，使用 LinkedHashMap 的构造函数将 accessOrder 设置为 true；
  3. 节点多于 MAX_ENTRIES ，移除链表首部的节点，覆盖 removeEldestEntry() 方法实现；
