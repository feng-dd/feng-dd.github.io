发表于 2019-05-08 | 更新于: 2019-05-08 | 分类于 JVM 虚拟机 | | 热度: ℃

JVM

对于 Java 程序员来说，在虚拟机自动内存管理机制下，不容易出现内存泄漏和内存溢出问题。
正是因为 Java 程序员把内存控制权利交给 Java 虚拟机，一旦出现内存泄漏和溢出方面的问题，
如果不了解虚拟机是怎样使用内存的，那么排查错误将会是一个非常艰巨的任务。所以对 Java 虚拟机的了解是必须的。

Java 运行时数据区(Java内存区域)

Java 虚拟机在运行 Java 程序时，会将它管理的区域划分为若干个不同的内存区域。
JDK 1.8 与之前版本略有区别。


1.程序计数器

程序计数器，线程私有的，可看做当前线程执行的字节码的行号指示器，字节码解释器通过改变计数器的值来选取下一条执行的字节码指令，
分支，循环，跳转，异常处理，线程恢复等都依赖于它。

为了在线程切换时，能恢复正确的执行顺序，每个线程都有一个独立的程序计数器，保证各线程之间互不影响，所以它是线程私有的内存区域。

线程计数器的作用有：1.字节码解释器通过它实现代码的流程控制。
2.多线程在切换时，保证当前线程切换回来后，正确执行。

内存区域中，唯一不会出现 OutOfMemoryError；生命周期与线程相同。

2.虚拟机栈

虚拟机栈，线程私有的，是 Java 方法执行的内存模型，栈是一个后进先出的数据结构，由一个个栈帧组成，方法的执行过程就是一个栈帧入栈到出栈的过程。

栈帧存储了局部变量表，操作数栈，动态链接，方法出口信息；每一个方法的调用对应一个栈帧被压入栈中，每一个方法调用结束，对应的栈帧被弹出栈并释放内存空间。
Java方法的两种返回方式：return、抛出异常；无论哪种方式，都会使一个栈帧被弹出。

局部变量表主要存放基本数据类型、对象引用。

虚拟机栈会出现两种异常 StackOverFlowError、OutOfMemoryError;
StackOverFlowError：当虚拟机栈内存不允许动态扩展时，线程的请求大小超过了虚拟机栈内存大小，就会抛出该异常。
OutOfMemoryError：当虚拟机栈内存允许动态扩展时，虚拟机在动态扩展时无法申请到足够的内存时，就会抛出该异常。

线程私有的，生命周期和线程相同，可能出现 StackOverFlowError、OutOfMemoryError。
可以通过 -Xss 这个虚拟机参数指定每个线程的Java虚拟机栈内存大小 java -Xss512M HackTheJava

3.本地方法栈

与虚拟机栈几乎相同，区别在于虚拟机栈为虚拟机执行 Java 方法服务，本地方法栈是为 Native 方法服务的。
HotSpot 虚拟机将俩者合二为一。

4.堆

堆是 Java 虚拟机管理的内存最大的区域，堆是线程共享的内存区域，在虚拟机启动时创建，存放对象实例及数组。

堆是垃圾收集器管理的主要区域，也叫 GC 堆；现代垃圾收集器基本采用的都是分代收集算法，按此，可以将堆分为新生代和老年代。
新生代还可以细分为 Eden 区、From Survivor、To Survivor，目的是根据对象特征，更好的分配和回收内存。

对象一般都会分配到 Eden 区，在进行一次新生代回收(Minor GC)后，如果对象还存活，对象进入 s0 区 或 s1 区，对象年龄+1，
当对象年龄到达一定程度(默认15)，对象晋升老年代。

堆可以处于物理上不连续的内存空间，只要逻辑上是连续的即可，与磁盘一样；当堆内存已经用完且无法动态扩展时，会触发 OutOfMemoryError。

线程共享，生命周期与虚拟机相同，可能出现 OutOfMemoryError 。
对象晋升到老年代的年龄阈值，可以通过参数 -XX:MaxTenuringThreshold 来设置。
可以通过 -Xms 和 -Xmx 这俩个虚拟机参数指定一个程序的堆内存大小的初始值和最大值 java -Xms1M -Xmx2M HackTheJava。

5.方法区

方法区是线程共享的内存区域，用于存放虚拟机已加载的类信息、常量、静态变量、即时编译器编译后的的代码。

Java 虚拟机规范将方法区划分为堆的逻辑区域，为了与堆区分，称为 Non-Heap(非堆)，与堆相同，可以出于物理上不连续，但逻辑上连续的内存空间。

HotSpot 将方法区称作”永久代”。垃圾会收集器很少回收方法区，但是进入方法区的数据并不一定是永久存在的。

JDK 1.8 将方法区移除，取而代之的是元空间，处于直接内存中， -XX:MetaspaceSize 可以指定元空间的大小,
与方法区不同，如果不指定元空间的大小，随着加载的类越来越多，虚拟机会耗尽所有可用的系统内存。

6.运行时常量池

运行时常量池是方法区的一部分，用于存放编译器生成的各种字面量和符号引用,当常量池的无法向方法区申请到更多的内存时，会触发OutOfMemoryErr。

字面量：字符串、基本类型的常量

JDK 1.7 将运行时常量池放入了堆内存。

7.直接内存

不属于Java 虚拟机内存，这部分内存也被频繁使用，当 Java 虚拟机内存大小 + 直接内存大小 > 物理内存大小，会触发 OutOfMemoryErr。

NIO 可以直接操作直接内存，在一些场景中避免了数据在 Java 堆与 Native 堆来回复制。也因此很容易操作 OutOfMemoryErr。

类加载与对象创建的过程


对象创建过程

1.虚拟机碰到一条 New 指令，触发对象创建，根据指令参数尝试定位运行时常量池中该类的符号引用，
并检查该类是否被加载、解析、初始化过。如果没有，则先进行该类加载。
2.类的加载，将类信息加载到方法区的过程中，静态内容(静态变量、静态代码块、静态方法)加载到静态区。

①通过类加载器(ClassLoader)将字节码文件(.class)加载到虚拟机(JVM)。

②虚拟机对字节码文件进行验证，过滤掉不符合虚拟机编译规则的文件，确保类加载的正确性。

③在方法区中，给类的静态变量分配内存，默认初始化赋予零值(数据的系统默认值)，准备好类的静态信息。

④将运行时常量池中的类的符号引用解析为直接引用(直接指向目标的引用指针)。

⑤将类的静态变量显示初始化(赋予正确的初始值)

⑥显示初始化之后，开始执行静态的代码块。先执行父类的静态代码块，再执行子类的静态代码块。

3.堆中为对象分配内存空间，大小在类加载时已知，分配方式有俩种：指针碰撞和空闲列表。
选择哪种方式分配取决于堆内存是否规整，堆内存规整取决于垃圾回收的算法是”标记-整理”还是”标记-清除”。

①指针碰撞:适用于堆内存规整(没有内存碎片),已使用的内存和空闲的内存间有一个分界值指针，
通过指针向未使用内存移动对象内存大小位置即可。GC收集器:Serial收集器(单线程)、ParNew收集器(多线程)。

②空闲指针:适用于堆内存不规整，虚拟机维护了一个列表，记录了空闲的内存块，
划分一块足够大的内存给对象，并跟新列表记录。

4.分配到内存的对象进行实例变量赋零值。

5.设置对象头，对象的元数据、对象的哈希码、对象的年龄、锁状态等。

6.执行初始化，构造函数调用(入栈)，显示初始化实例变量，构造函数执行完毕(出栈)，将对象分配的内存地址赋给引用对象。

调用子类构造时，默认第一行隐式调用 super()(父类构造)，若第一行显示调用 this()(子类构造),最后还是会先调用到另一个构造里的 super()。

创建对象的方式有多种:new 关键字、反射(Class对象或构造对象.newInstance())、clone()、反序列化
